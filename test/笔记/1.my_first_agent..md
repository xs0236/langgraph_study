# 第二步

为什么“忘了”打八折？
你发现第二次问“那上海的呢？”，Agent 只回答了天气，没再计算打折。这其实不是记忆丢失，而是 Agent 的逻辑判断结果。
让我们看看它的“思考过程”（就是你加的 DEBUG 日志）：
第一次对话：
你问：“帮我查一下北京的天气。如果打八折是多少”
Agent 思考：用户问了天气，还要求计算。好，我先调用 get_weather 查到是 25 度，再调用 calculator 算一下 25 \* 0.8。
结论： 这是一个明确的、一次性的复合指令。
第二次对话：
你问：“那上海的呢？”
Agent 思考：用户问的是另一个城市的天气。我需要调用 get_weather 查上海的天气。至于“打八折”？那是上一轮对话的特定要求，用户这次没提，我就不需要再算了。
结论： 它认为“打八折”这个任务已经随着上一轮对话结束了。
简单来说，Agent 记得所有对话历史，但它不会把上一轮的“具体任务”自动延续到下一轮，除非你明确要求。
🧪 如何验证记忆是有效的？
你可以用一个更直接的问题来测试它的记忆。在第二次对话时，不要问“上海的天气”，而是问一个必须依赖上文才能回答的问题。
测试问题：
“刚才北京打完折之后是多少度来着？”
预期结果：
Agent 应该能直接从记忆中找到“20度”这个信息并回答你，而不需要再次调用任何工具。因为它只是在复述历史对话中的事实。
你可以试试看，这能很好地证明它的记忆功能是正常工作的！

# 第三步

🚀 进入下一步：人工干预 (Human-in-the-loop)
接下来，我们可以学习 LangGraph 最强大的功能之一：人工干预。
这个功能可以让 Agent 在执行关键操作（比如发送邮件、删除文件、支付订单）之前先暂停下来，等待你的批准。这在实际应用中至关重要，可以防止 Agent “闯祸”。
核心思路
中断：在编译工作流时，指定一个“中断点”。当流程走到这个节点之前，会自动暂停。
审查：程序暂停后，你可以检查当前的状态，决定是继续还是修改。
恢复：你给出指令后，程序再继续执行。

# 第四步

接下来学什么？
到这里，你已经具备了构建一个工业级 Agent 的所有基础组件：
状态管理 (State)：Agent 的记忆。
工具调用 (Tools)：Agent 的手脚。
条件路由 (Conditional Edges)：Agent 的逻辑判断。
持久化记忆 (Checkpointer)：跨轮次的上下文记忆。
人工干预 (Interrupts)：安全控制机制。
下一步，我建议我们进入企业级 Agent 的进阶架构。在实际业务中，我们很少只用一个大模型“包揽所有活”，而是会用到以下两种高级模式：
方向一：多 Agent 协作 (Multi-Agent)
想象一个场景：用户问“帮我订一张明天去北京的机票，并查一下那边的天气”。
Agent A (调度员)：负责理解用户意图，把任务分发下去。
Agent B (天气专家)：只负责查天气。
Agent C (票务专家)：只负责查机票。
Agent A (调度员)：收集 B 和 C 的结果，整合成一句话回复用户。
方向二：自定义状态机 (Custom State Machine)
目前的 Agent 是自由发挥的（ReAct 模式），但在很多业务中，流程是固定的。比如“客服工单处理”：
节点1：收集客户信息。
节点2：判断问题类型。
节点3：如果是退款，走退款流程；如果是投诉，走安抚流程。
这种严格受控的图结构，是 LangGraph 区别于其他框架的最大优势。
