你刚才运行的代码是 LangGraph 最经典的 "ReAct Agent"（推理+行动） 模式。为了继续深入学习，我们不应该只停留在“跑通代码”，而是要理解它为什么能工作，并尝试增加一点复杂度。
接下来我为你准备了 3 个进阶学习步骤，你可以按顺序尝试：
第一步：观察“思维链” (Trace the Logic)
你现在的代码里有一个 print 语句在工具函数里。
现象： 当 LLM 决定调用工具时，它会先返回一个特殊的消息（包含 tool_calls），而不是直接回答。
任务： 修改 chatbot 函数，打印出 LLM 的原始响应对象，看看它在调用工具前到底“想”了什么。
修改建议：
在你的 chatbot 函数中，把 return 之前的部分改成这样：
python

编辑

def chatbot(state: State):
response = model.invoke(state["messages"])

    # 【新增】打印 LLM 的原始决策
    print(f"[DEBUG] LLM 回复内容: {response.content}")
    print(f"[DEBUG] LLM 想要调用的工具: {response.tool_calls}")

    return {"messages": [response]}

运行后观察： 你会发现 LLM 会先输出空的内容（content为空），但 tool_calls 里会有 get_weather 和参数 {'city': '北京'}。这就是 Agent "思考" 的过程。
第二步：增加“记忆” (Add Memory)
目前的 Agent 是“金鱼记忆”，它不记得上一轮对话。如果你接着问“那上海呢？”，它可能会不知道你在问天气。
LangGraph 使用 Checkpointer 来实现持久化记忆。
操作步骤：
引入 Checkpointer：
python

编辑

from langgraph.checkpoint.memory import MemorySaver

# 初始化一个内存存储器

memory = MemorySaver()
编译时传入存储器：
将原本的 app = workflow.compile() 改为：
python

编辑

app = workflow.compile(checkpointer=memory)
运行时加入配置：
在调用 app.stream 时，需要告诉它这是“谁”的对话（Thread ID）。
python

编辑

config = {"configurable": {"thread_id": "user_001"}}

# 第一次提问

inputs = {"messages": [("user", "你好，帮我查一下北京的天气")]}
for output in app.stream(inputs, config=config): # ... (保持原有的打印逻辑)
pass

# 第二次提问 (测试记忆)

inputs2 = {"messages": [("user", "那上海的呢？")]} # 注意：这里不需要传历史消息，Agent会自动从 memory 读取
print("\n--- 第二轮对话 ---")
for output in app.stream(inputs2, config=config): # ...
pass
预期结果： 即使你在第二次输入时没有提“天气”两个字，Agent 也能结合上下文知道你要查上海的天气。
第三步：增加一个真实工具 (Real Tool)
现在的 get_weather 是写死的假数据。我们可以加一个简单的计算器工具，这能体现 LLM 处理逻辑短板的能力（LLM 数学通常不好，但调用计算器很准）。
代码添加：
python

编辑

# 定义一个新的工具

@tool
def calculator(expression: str) -> float:
"""计算数学表达式的结果，例如 '2 + 2' 或 '15 \* 4'"""
try: # 注意：在生产环境中 eval 是不安全的，但在本地学习 demo 中可以使用
result = eval(expression)
return result
except Exception as e:
return f"计算错误: {e}"

# 更新工具列表

tools = [get_weather, calculator]

# 重要：更新模型的绑定

model = ChatOpenAI(model="glm-4-flash", temperature=0).bind_tools(tools)

# 注意：如果你用的是 GLM，确保模型名称是你刚才调试通过的那个
