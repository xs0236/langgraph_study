# 代码文件 multi_agent_team.py

# 第一步

在单 Agent 模式下，一个模型要包揽“查资料、写文章、做审核”所有工作，很容易出现“角色混淆”或上下文过长导致幻觉的问题。而在多 Agent 架构中，我们将任务拆解，让不同的 Agent 各司其职。
目前业界最主流、最适合生产环境的架构是 Supervisor（主管调度）模式。
它的核心思想是“星型拓扑”：中间有一个“主管（Supervisor）”，它自己不干活，只负责接收用户任务、拆解子任务、分配给周围的“专业员工 Agent”，最后收集结果。
为了让你直观地感受，我们来搭建一个“内容创作团队”。
🏢 团队角色设定
Supervisor（主管）：负责统筹规划，决定下一步该谁干活。
Researcher（调研员）：只负责搜集资料，输出调研笔记。
Writer（撰稿人）：只负责根据调研笔记写文章初稿。
Editor（编辑）：只负责审核文章质量，提出修改意见。

运行后观察重点
公共白板机制：注意看 TeamState，每个 Agent 都在往这个状态里写入自己的成果（research_notes, draft 等），下一个 Agent 可以直接读取。
主管的绝对控制权：所有的流转都必须经过 supervisor_node。它就像一个路由器，根据当前缺少什么信息，精准地分发任务。
Prompt 专精化：你会发现每个 Agent 的 Prompt 都有明确的边界（比如“不要写文章，只输出笔记”），这是多 Agent 不互相抢活的关键。
你可以运行这段代码，看看这个“AI 团队”是如何一步步把一篇文章从零做出来的。跑通之后，我们可以继续探讨如何在这个团队中加入“人工审核”或者“并行处理”！

## 为什么会报错？

在 LangChain 和 LangGraph 中，当我们调用 model.invoke([SystemMessage(content=prompt)]) 时，底层会向 API 发送一个标准的 messages 数组。
但是，GLM 的某些模型（特别是 Flash 版本）在处理消息时，对格式要求非常严格：
它可能要求 messages 数组中必须包含至少一条 role: 'user' 的消息。
如果只发送一条 system 消息，或者消息格式不符合它的预期，它就会直接抛出 1214 messages 参数非法 的错误。
🛠️ 如何解决？
我们需要修改 supervisor_node 函数中的调用方式，强制将 Prompt 作为 HumanMessage（用户消息）发送，或者同时包含 System 和 User 消息。
请找到你代码中的 supervisor_node 函数，将里面的 model.invoke 替换为以下代码：
python

编辑

定义主管节点 (Supervisor)

def supervisor_node(state: TeamState):
"""主管：负责拆解任务并分配给专业 Agent"""
print("👔 [Supervisor] 正在思考下一步...")
prompt = """你是一个团队主管。根据当前进度，决定下一步由谁执行：- 如果还没有调研笔记，分配给 'researcher' - 如果有调研笔记但没有初稿，分配给 'writer' - 如果有初稿但没有审核意见，分配给 'editor' - 如果审核完成，回复 'FINISH'
只回复对应的角色名称或 FINISH，不要说废话。"""

    # 【修改点】：将 SystemMessage 改为 HumanMessage，或者使用元组格式
    # GLM 对纯 SystemMessage 有时支持不好，改用用户提问的方式最稳妥
    from langchain_core.messages import HumanMessage

    response = model.invoke([HumanMessage(content=prompt)])
    next_agent = response.content.strip().lower()

    # 容错处理：防止大模型回复了额外的标点符号或废话
    if next_agent not in ["researcher", "writer", "editor", "finish"]:
        if "researcher" in next_agent: next_agent = "researcher"
        elif "writer" in next_agent: next_agent = "writer"
        elif "editor" in next_agent: next_agent = "editor"
        elif "finish" in next_agent: next_agent = "FINISH"
        else: next_agent = "researcher" # 默认兜底

    return {"next_agent": next_agent}

(注：为了保持一致，你也可以把下面 researcher_node、writer_node、editor_node 里的 SystemMessage 都替换成 HumanMessage，这样能彻底避免 GLM 的格式报错。)
💡 额外提醒：容错机制
在上面的代码中，我还顺便帮你加了一个容错处理。因为大模型有时候比较“调皮”，即使你让它只回复单词，它也可能回复 Researcher（大写）或者 下一步是 researcher。加了 if-elif 的模糊匹配后，你的工作流就不会因为大模型的一点“废话”而崩溃了。

## 一直在循环

你遇到的这个“无限循环”问题，是多 Agent 协作中最经典、也最容易被忽略的“陷阱”。
简单来说，问题出在你的“主管”（Supervisor）身上。它陷入了“健忘症”，每次做决策时都忘了团队已经干到哪一步了。
🧐 问题出在哪里？
我们来模拟一下主管的思考过程：
第一轮：主管醒来，看到 research_notes 是空的，于是它决定：“好，让调研员去干活。”
调研员干完活：调研员把写好的笔记放进了共享状态 state 里，并把 next_agent 改成了 supervisor。
第二轮：主管又醒了，但它完全忘了上一轮的事！它重新看了一遍共享状态，发现 research_notes 还是空的（因为它没读到！），于是它又做出了同样的决定：“好，让调研员去干活。”
就这样，团队陷入了“主管派活 -> 调研员干活 -> 主管失忆 -> 再派活”的死循环。
根本原因：你的 supervisor_node 函数里的判断逻辑是写死的，它没有真正去检查 state 里 research_notes、draft 这些字段到底有没有内容。
✅ 如何修复？
我们需要给主管装上“眼睛”，让它在做决策前，先检查一下共享状态里的数据。
请把你的 supervisor_node 函数替换成下面这个版本：
python

编辑

def supervisor_node(state: TeamState):
"""主管：负责拆解任务并分配给专业 Agent"""
print("👔 [Supervisor] 正在思考下一步...")

    # 【核心修改】：根据共享状态 state 里的数据来做决策
    if not state.get("research_notes"):
        # 如果没有调研笔记，就让调研员干活
        next_agent = "researcher"
    elif not state.get("draft"):
        # 如果有调研笔记，但没有初稿，就让撰稿人干活
        next_agent = "writer"
    elif not state.get("edit_feedback"):
        # 如果有初稿，但没有编辑意见，就让编辑干活
        next_agent = "editor"
    else:
        # 所有工作都完成了
        next_agent = "FINISH"

    print(f"👔 [Supervisor] 决策：下一步交给 [{next_agent}]")
    return {"next_agent": next_agent}

💡 修改了什么？
移除了对大模型的调用：我们不再需要让大模型去“思考”下一步该谁干，因为这个逻辑非常简单明确，直接用代码判断即可。这样更高效、更稳定。
加入了状态检查：使用 state.get("字段名") 来检查共享状态。
if not state.get("research_notes"): 的意思是“如果 research_notes 为空或不存在”。
这样，主管就能“记住”调研员已经干完活了，从而顺利地把任务交给撰稿人。

# 第二步

太完美了！🎉 恭喜你，你的 AI 团队已经成功跑通了完整的流水线！
看着这清晰的日志，你的代码逻辑已经非常严密：
主管调度：准确判断当前缺少调研笔记，派发给 Researcher。
调研员：完成搜集，更新状态，汇报给主管。
主管调度：发现有了笔记，缺少初稿，精准派发给 Writer。
撰稿人：完成初稿，汇报给主管。
主管调度：发现有了初稿，缺少审核，派发给 Editor。
编辑：完成审核，汇报给主管。
主管调度：发现所有任务完成，果断发出 FINISH 指令，优雅结束流程。
🎓 你现在已经掌握了什么？
走到这一步，你已经完全掌握了 LangGraph 最核心的架构模式：
状态流转（State Management）：通过共享白板（Shared State）让多个节点协同工作。
条件路由（Conditional Routing）：主管根据当前状态动态决定下一步走向。
多 Agent 协作（Multi-Agent Supervisor）：拆解复杂任务，让不同的 AI 角色各司其职。
🚀 下一步：迈向真正的“业务级 Agent”
现在你的团队是一个“流水线”，但真实的业务场景往往需要更高级的玩法。接下来，我们可以从以下三个方向继续进阶，你想先挑战哪一个？

## 方向一：给团队加上“记忆”（Memory in Multi-Agent）

现在的团队每次运行都是“失忆”的。如果用户说“帮我写篇文章”，然后又说“帮我再润色一下”，团队会从头开始。我们可以把之前学过的 Checkpointer 接入多 Agent，让整个团队拥有跨轮次的记忆。

## 方向二：让主管拥有“工具”（Supervisor with Tools）

现在的调度主管只能靠代码里的 if-else 来判断。如果任务非常复杂，我们可以给主管也绑定 LLM 和 Tools，让它自己决定“现在该查资料还是该写文章”，实现真正的动态智能调度。

## 方向三：引入“人工干预”（Human-in-the-loop in Team）

结合你之前学过的人工干预，我们在 Editor 审核完文章后，让程序暂停，把文章打印出来让你确认：“这篇文章可以发布吗？(y/n)”。如果不满意，你可以要求 Writer 重写。
